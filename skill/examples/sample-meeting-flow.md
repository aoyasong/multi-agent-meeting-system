# 示例会议流程

## 示例一：头脑风暴 - 产品命名

### 输入

```
用户：我们来头脑风暴一下新AI产品的命名
```

### 执行流程

**Step 1: 会议创建**

```
主Agent:
  识别会议类型：brainstorm
  会议参数：
    - title: "AI产品命名头脑风暴"
    - participants: [产公公, 技公公, 设公公]
    - type: brainstorm
    - votingComplexity: simple

输出：
> 创建头脑风暴会议...
>
> **会议主题**：AI产品命名头脑风暴
> 参会者：产公公、技公公、设公公
> 规则：自由发言，数量优先，不批判
>
> 开始会议？
```

**Step 2: 发散阶段**

```
用户: 确认开始

主Agent:
  meeting_start(meetingId)

输出：
> 🎯 头脑风暴开始！
> 主题：AI产品命名
> 请自由提出命名创意，数量优先！

[开放自由发言模式]

产公公: 我提议叫 "智言" —— 体现智能表达
技公公: "OpenMind" 怎么样 —— 开放思维的AI
设公公: "ClawGPT" —— 品牌关联
技公公: 或者 "BrainWave" —— 思维波
产公公: "灵感爪" —— 中性且有趣
设公公: "SymphonyAI" —— 思维交响曲

主Agent:
  recording_take_note() 记录每个建议

输出：
> 收集中... 当前 6 个命名：
> 1. 智言 - 产公公
> 2. OpenMind - 技公公
> 3. ClawGPT - 设公公
> 4. BrainWave - 技公公
> 5. 灵感爪 - 产公公
> 6. SymphonyAI - 设公公
>
> 还有补充吗？10秒后进入收敛阶段。
```

**Step 3: 收敛阶段**

```
主Agent:
  归类整理 → 创建投票

输出：
> 发散阶段结束，共收集 6 个命名。
>
> 归类：
> **品牌关联**
> - ClawGPT
> - 灵感爪
>
> **英文风格**
> - OpenMind
> - BrainWave
> - SymphonyAI
>
> **中文风格**
> - 智言
>
> 启动优先级投票，每人选 Top 2。

主Agent:
  voting_create(
    meetingId,
    topic="命名优先级",
    options=[6个命名],
    complexity="simple",
    multiSelect=true,
    maxSelections=2
  )

[等待 3 分钟投票窗口]

产公公: 我投票 "智言" 和 "ClawGPT"
技公公: 投票 "BrainWave" 和 "OpenMind"
设公公: 投票 "ClawGPT" 和 "SymphonyAI"
```

**Step 4: 结果宣布**

```
主Agent:
  voting_get_result()

输出：
> 🗳️ 投票结果（3/3 已投票）
>
> 🥇 ClawGPT - 2票
> 🥈 智言 - 1票
> 🥉 BrainWave - 1票
> 🥉 OpenMind - 1票
> 🥉 SymphonyAI - 1票
>
> 建议：采用 "ClawGPT" 作为产品名称。
>
> 是否需要运行决胜轮解决并列？
```

**Step 5: 结束与输出**

```
用户: 不用了，就ClawGPT吧

主Agent:
  output_generate_summary()
  output_generate_action_items()
  meeting_end()

输出：
> 头脑风暴会议结束。
>
> ✅ 命名确定：**ClawGPT**
>
> 纪要已生成：
> - 候选命名 6 个
> - 可选备选：智言、BrainWave
>
> 行动项：
> - [ ] 品牌设计确认 - 设公公
> - [ ] 域名注册评估 - 技公公
```

## 示例二：需求评审 - 积分系统

### 输入

```
用户: 评审一下积分系统PRD
```

### 执行流程

**Step 1: 会议创建**

```
主Agent:
  识别会议类型：requirement-review
  加载PRD文档
  生成议程

输出：
> 创建需求评审会议...
>
> **评审对象**：用户积分系统 PRD v1.0
> 参会者：产公公、技公公、设公公、QA公公
> 评审维度：完整性、可行性、价值、风险
>
> 议程：
> 1. 需求陈述 - 5分钟
> 2. 分项审议 - 25分钟
> 3. 综合评审 - 10分钟
>
> 开始评审？
```

**Step 2: 需求陈述**

```
主Agent:
  meeting_start()
  speaking_grant(产公公)

输出：
> 需求评审开始。
>
> 请产公公做需求陈述。

产公公:
  积分系统目标：提升用户活跃度 15%
  核心功能：
  1. 积分获取规则 - 签到、任务、消费
  2. 积分消费渠道 - 兑换、抵扣、抽奖
  3. 积分过期机制 - 年度清零，提醒
  非功能需求：TPS 1000+，一致性保障

主Agent:
  recording_take_note(需求要点)
```

**Step 3: 分项审议**

```
主Agent:
  议题1: 积分获取规则

输出：
> 📋 当前审议：**积分获取规则**
>
> 功能描述：
> - 签到：每日5分
> - 任务：完成即得，10-50分
> - 消费：1元=1分
>
> 请技术负责人评估可行性。

技公公:
  技术可行。建议：
  1. 高并发场景用消息队列解耦
  2. 积分计算用缓存+定时落库
  3. 防刷机制需要配合风控

主Agent:
  recording_take_note(技术意见)

输出：
> 记录技术建议：
> - MQ解耦积分计算
> - 缓存优化性能
> - 接入风控
>
> 功能项：积分获取规则 ✅ 通过
>
> 📋 下一项：**积分消费渠道**

[继续逐项审议...]

主Agent:
  议题2: 积分消费渠道

设公公:
  消费渠道设计建议：
  - 商城页面需要清晰的积分价值展示
  - 抵扣场景要考虑用户体验
  建议：首期仅支持商城兑换，后续迭代

技公公:
  商城兑换技术复杂度高，建议先做"积分抵扣"
  优先级：抵扣 > 商城 > 抽奖

主Agent:
  记录讨论结果：
  - 首期聚焦：积分抵扣
  - 商城作为迭代二期

输出：
> 功能项：积分消费渠道 ⚠️ 修改后通过
> 调整：首期仅做积分抵扣
```

**Step 4: 综合评审**

```
主Agent:
  汇总评审结果

输出：
> 📊 评审汇总
>
> | 功能项 | 评估 | 备注 |
> |--------|------|------|
> | 积分获取规则 | ✅ | 需配合风控 |
> | 积分消费渠道 | ⚠️ | 首期仅抵扣 |
> | 积分过期机制 | ✅ | - |
> | 非功能需求 | ✅ | 需压测验证 |
>
> 整体决议投票启动。

主Agent:
  voting_create(
    meetingId,
    topic="需求整体决议",
    options=["通过", "修改后通过", "驳回", "待定"],
    complexity="moderate"
  )

[投票进行...]

结果: 修改后通过 (4/4同意)
```

**Step 5: 结束与输出**

```
主Agent:
  output_generate_summary()
  output_generate_action_items()
  meeting_end()

输出：
> ✅ **评审结论：修改后通过**
>
> 条件：
> 1. 首期范围聚焦积分抵扣
> 2. 需完成风控方案设计
>
> 行动项：
> | 行动 | 负责人 | 截止 |
> |------|--------|------|
> | 风控方案设计 | 技公公 | 周五 |
> | 更新PRD范围 | 产公公 | 明天 |
> | 设计积分抵扣流程 | 设公公 | 周三 |
>
> 纪要已生成。
```

## 示例三：技术评审 - 微服务方案

### 输入

```
用户: 评审微服务架构方案
```

### 执行流程（简化版）

```
主Agent:
  创建技术评审会议
  参会者：技公公、架构师Agent、运维Agent

输出：
> 🔍 技术评审开始
>
> 评审对象：微服务架构改造方案
> 评审维度：架构、扩展性、性能、安全、可维护

技公公: [方案陈述]
  当前问题：单体扩展瓶颈
  方案：拆分5服务 + K8s + Service Mesh
  技术栈：Spring Cloud + Istio
  预期收益：独立部署、弹性伸缩

主Agent:
  多维评估

[架构评估]
架构师Agent: 服务拆分合理，建议API Gateway统一入口
评估结果: ✅ 架构合理性通过

[扩展性评估]
技公公: 水平扩展支持，预估支撑10x流量
评估结果: ✅ 可扩展性通过

[性能评估]
运维Agent: 链路追踪存在开销，建议灰度验证
评估结果: ⚠️ 性能需验证

[安全评估]
架构师Agent: 服务间mTLS + JWT方案完善
评估结果: ✅ 安全性通过

[可维护性评估]
技公公: 完善监控告警，日志统一ELK
评估结果: ✅ 可维护性通过

主Agent:
  汇总风险评估

输出：
> ⚠️ 风险项：性能（Istio开销）
>
> 建议：先PoC验证链路性能
> 应对：压测对比，必要时优化Sidecar

[决议投票]
voting_create(meetingId, "技术方案决议")

结果: **修改后通过**
条件：完成PoC性能验证
```

[议题2: 团队分工]

- 产品：产公公

## 流程总结

| 阶段 | 时长 | 关键操作 |
|------|------|----------|
| 创建 | 1分钟 | 识别类型、确认参数 |
| 开场 | 1-5分钟 | 规则说明、议程展示 |
| 主体 | 15-60分钟 | 按议程推进、记录要点 |
| 投票 | 3-10分钟 | 收集投票、结果统计 |
| 结束 | 2-5分钟 | 生成纪要、行动项确认 |
