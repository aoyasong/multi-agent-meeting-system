# 技术评审场景

## 场景说明

**适用情况**：
- 技术方案评审
- 架构设计评审
- 技术选型决策
- 重大技术变更审批

**预期产出**：
- 技术方案通过/修改决议
- 风险清单和应对建议
- 技术债务识别

## 会议参数建议

```json
{
  "type": "technical-review",
  "votingComplexity": "moderate",
  "speakingMode": "structured",
  "estimatedDuration": "45-90 minutes"
}
```

## 标准流程

### 1. 开场（2分钟）

主Agent发言：
> 技术评审会议开始
>
> **评审对象**：{技术方案名称}
> **方案提出者**：{提出者}
> **评审维度**：架构合理性、可扩展性、性能、安全性、可维护性
>
> 参会者：{列表}
> 预计时长：{N}分钟
>
> 请方案提出者进行技术方案介绍。

### 2. 方案陈述（10-15分钟）

**发言人**：技术方案提出者（通常是技公公）
**内容结构**：
- 背景与问题定义
- 方案概述
- 技术架构图（如有）
- 关键技术决策
- 风险评估

**主Agent操作**：
```
speaking_grant(meetingId, proposerId)
recording_take_note(meetingId, {
  type: "proposal",
  summary: "方案核心内容..."
})
```

### 3. 多维度评审（25-45分钟）

按评审维度逐项审议：

#### 3.1 架构合理性

主Agent发言：
> 📐 **维度一：架构合理性**
>
> 评估要点：
> - 模块划分是否清晰
> - 职责边界是否明确
> - 依赖关系是否合理
>
> 请各参会者发表意见。

**结构化发言顺序**：
1. 架构师/资深技术 → 架构评价
2. 相关模块负责人 → 接口影响
3. 其他技术参会者 → 自由提问

**评估记录**：
```
架构合理性：✅ 通过 / ⚠️ 有建议 / ❌ 有风险
- 评价：{具体意见}
- 建议/风险点：{...}
```

#### 3.2 可扩展性

主Agent发言：
> 📈 **维度二：可扩展性**
>
> 评估要点：
> - 是否支持水平扩展
> - 未来需求变化的适应性
> - 是否存在扩展瓶颈

**评估记录**：
```
可扩展性：✅ 通过 / ⚠️ 有建议 / ❌ 有风险
- 水平扩展：{评估结果}
- 瓶颈识别：{...}
```

#### 3.3 性能评估

主Agent发言：
> ⚡ **维度三：性能**
>
> 评估要点：
> - 响应时间预期
> - 吞吐量预估
> - 资源消耗评估
> - 是否存在性能热点

**评估记录**：
```
性能评估：✅ 通过 / ⚠️ 有建议 / ❌ 有风险
- 预期TPS：{数值}
- 响应时间：{数值}
- 热点风险：{...}
```

#### 3.4 安全性

主Agent发言：
> 🔒 **维度四：安全性**
>
> 评估要点：
> - 认证授权机制
> - 数据安全（加密、脱敏）
> - 接口安全（限流、防注入）
> - 敏感操作审计

**评估记录**：
```
安全性评估：✅ 通过 / ⚠️ 有建议 / ❌ 有风险
- 认证授权：{评估}
- 数据安全：{评估}
- 风险点：{...}
```

#### 3.5 可维护性

主Agent发言：
> 🔧 **维度五：可维护性**
>
> 评估要点：
> - 代码可读性要求
> - 日志与监控方案
> - 文档完整性
> - 技术债务评估

**评估记录**：
```
可维护性评估：✅ 通过 / ⚠️ 有建议 / ❌ 有风险
- 代码规范：{要求}
- 监控方案：{描述}
- 技术债务：{识别的债务}
```

### 4. 风险汇总与应对（10分钟）

主Agent发言：
> 评审汇总：
>
> | 维度 | 评估 | 关键问题 |
> |------|------|----------|
> | 架构合理性 | ✅ | - |
> | 可扩展性 | ⚠️ | {问题} |
> | 性能 | ✅ | - |
> | 安全性 | ❌ | {风险} |
> | 可维护性 | ✅ | - |
>
> 需重点讨论的风险项：{风险项}
>
> 请提出应对建议。

**应对方案讨论**：
- 现有方案调整
- 增加补救措施
- 分阶段解决

### 5. 决议投票（5-10分钟）

**创建投票**：
```
voting_create(
  meetingId,
  topic="技术方案决议",
  options=[
    "通过",
    "修改后通过（有条件）",
    "不通过-需重新设计",
    "待定-需PoC验证"
  ],
  complexity="moderate"
)
```

### 6. 结束与输出

**决议宣布**：
> **技术评审结论**：{决议}
>
> {决议说明}
>
> 待解决的遗留项：
> 1. {遗留项1} - 责任人：{Agent} - 截止：{日期}
> 2. {遗留项2} - 责任人：{Agent} - 截止：{日期}
>
> 会议结束。技术评审纪要已生成。

**纪要结构**：

## 示例对话

**用户**：帮我组织一次技术评审，评审新的微服务架构方案

**主Agent**：
> 创建技术评审会议...
>
> **技术方案评审**：微服务架构改造
> 方案提出者：技公公
> 参会者：技公公、架构师Agent、运维Agent
> 预计时长：60分钟
>
> 确认开始？

**用户**：确认

**主Agent**：
> 技术评审开始。请技公公介绍方案。

**技公公**：
> 当前单体架构已难以支撑业务增长...
> 建议拆分为5个微服务：
> 1. 用户服务
> 2. 订单服务
> 3. 商品服务
> 4. 支付服务
> 5. 通知服务
>
> 技术栈：Spring Cloud + Kubernetes
> 预计改造周期：2个月
>
> 主要风险：
> - 分布式事务处理
> - 服务间调用链路追踪
> - 数据迁移一致性

**主Agent**：（记录完成）
> 方案介绍完毕，开始多维评审。
>
> 📐 **维度一：架构合理性**
>
> 架构师请发言。

**架构师Agent**：
> 服务拆分粒度合理。建议：
> 1. 用户服务可进一步拆分为认证服务和用户信息服务
> 2. 增加API Gateway层统一入口
>
> 整体架构合理。

**主Agent**：
> 记录：架构合理性 ✅ 通过
> 建议：考虑拆分用户服务、增加API Gateway
>
> 📈 **维度二：可扩展性**
> ...

**... 逐项评审 ...**

**主Agent**：
> 📊 **评审汇总**
>
> | 维度 | 评估 |
> |------|------|
> | 架构合理性 | ✅ |
> | 可扩展性 | ✅ |
> | 性能 | ⚠️ |
> | 安全性 | ⚠️ |
> | 可维护性 | ✅ |
>
> 需重点讨论：
> - 性能：分布式调用链路可能成为瓶颈
> - 安全：服务间认证方案需细化
>
> 请提出应对建议。

**技公公**：
> 性能问题：计划引入Service Mesh（Istio）优化链路
> 安全问题：采用mTLS + JWT双重认证

**主Agent**：
> 风险应对已确认。
>
> 启动决议投票。

**... 投票进行 ...**

**主Agent**：
> 投票结果：**修改后通过**
>
> 条件：
> 1. 补充Service Mesh方案详细设计
> 2. 完善安全认证方案文档
>
> 行动项：
> 1. [ ] Istio集成方案设计 - 技公公 - 截止：下周三
> 2. [ ] 安全方案文档化 - 技公公 - 截止：下周五
>
> 评审结束。

## 技术评审检查清单

### 架构检查
- [ ] 服务/模块划分合理
- [ ] 接口设计规范
- [ ] 数据流清晰
- [ ] 无循环依赖

### 性能检查
- [ ] 关键路径性能评估
- [ ] 缓存策略明确
- [ ] 数据库查询优化
- [ ] 异步处理合理

### 安全检查
- [ ] 认证授权完整
- [ ] 敏感数据加密
- [ ] API安全防护
- [ ] 审计日志完善

### 运维检查
- [ ] 部署方案可行
- [ ] 监控告警完整
- [ ] 容灾方案明确
- [ ] 回滚机制健全
