# 需求评审场景

## 场景说明

**适用情况**：
- 产品需求文档（PRD）评审
- 功能方案确认
- 跨团队需求对齐

**预期产出**：
- 需求通过/修改/驳回决议
- 待澄清问题清单
- 下一步行动项

## 会议参数建议

```json
{
  "type": "requirement-review",
  "votingComplexity": "moderate",
  "speakingMode": "structured",
  "estimatedDuration": "30-60 minutes"
}
```

## 标准流程

### 1. 开场（2分钟）

主Agent发言：
> 需求评审会议开始
>
> **评审主题**：{需求名称}
> **需求来源**：{提出者}
> **评审维度**：完整性、可行性、价值、风险
>
> 参会者：{列表}
> 预计时长：{N}分钟
>
> 请需求提出者先做 briefed。

### 2. 需求陈述（5分钟）

**发言人**：需求提出者（通常是产公公）
**内容**：
- 需求背景和目标
- 核心功能点
- 预期收益

**主Agent操作**：
- `speaking_grant(meetingId, proposerId)`
- 等待陈述完成
- `recording_take_note()` 记录关键点

### 3. 分项审议（20-40分钟）

按需求条目逐项审议：

#### 3.1 功能项审议模板

主Agent发言：
> 当前审议：**{功能项名称}**
> 描述：{功能描述}
>
> 请各参会者提出意见或疑问。

**发言顺序**（结构化模式）：
1. 技术负责人 → 技术可行性意见
2. 设计负责人 → 体验影响意见
3. 其他参会者 → 自由提问

**记录要点**：
```
功能项：{名称}
- 技术意见：{技公公的意见}
- 设计意见：{设公公的意见}
- 风险点：{识别的风险}
- 待澄清：{需要确认的问题}
- 结论：通过/修改后通过/待定
```

#### 3.2 冲突处理

当出现意见分歧时：

**主Agent**：
> 检测到对"{功能项}"存在不同意见：
> - A观点：{...}
> - B观点：{...}
>
> 建议：
> 1. 进一步讨论 → 延长3分钟
> 2. 投票决策 → 启动投票
> 3. 稍后线下处理 → 记录为待定
>
> 请选择处理方式。

### 4. 综合评审（5-10分钟）

#### 4.1 汇总待定项

主Agent发言：
> 需求评审汇总：
>
> ✅ 通过项：{N}个
> ⚠️ 修改项：{N}个
> ❓ 待定项：{N}个
>
> 待定项清单：
> 1. {待定项1} - 原因：{...}
> 2. {待定项2} - 原因：{...}

#### 4.2 整体决议投票

**创建投票**：
```
voting_create(
  meetingId,
  topic="需求整体决议",
  options=["通过", "修改后通过", "驳回", "待定-需更多信息"],
  complexity="moderate"
)
```

**投票窗口**：5分钟（中等复杂度）

### 5. 结束与输出

#### 5.1 决议宣布

主Agent发言：
> **评审结论**：{决议结果}
>
> {决议说明}
>
> 行动项：
> 1. [ ] {行动项1} - 负责人：{Agent} - 截止：{日期}
> 2. [ ] {行动项2} - 负责人：{Agent} - 截止：{日期}
>
> 会议结束。纪要将在会后发出。

#### 5.2 生成输出

**纪要结构**：
```markdown
## 需求评审纪要

**需求名称**：{名称}
**评审时间**：{时间}
**参会者**：{列表}
**决议**：{通过/修改后通过/驳回/待定}

### 需求概要
{需求简述}

### 评审详情

#### ✅ 通过项
1. {功能项} - 无异议
2. {功能项} - 已确认技术可行

#### ⚠️ 修改项
1. {功能项}
   - 问题：{描述}
   - 修改建议：{...}
   - 责任人：{Agent}

#### ❓ 待定项
1. {功能项} - 待澄清：{...}

### 行动项
| 序号 | 行动 | 负责人 | 截止日期 |
|------|------|--------|----------|
| 1 | ... | ... | ... |

### 后续安排
{下次评审时间/跟进方式}
```

## 示例对话

**用户**：评审一下PRD文档

**主Agent**：
> 收到需求评审请求。
>
> 请确认：
> - PRD文档位置：{路径}
> - 参会者：产公公、技公公、设公公
>
> 是否正确？

**用户**：确认，开始评审

**主Agent**：
> 需求评审会议开始
> **评审主题**：用户积分系统 v1.0
>
> 请产公公简述需求。

**产公公**：
> 积分系统目标是提升用户活跃度...
> 核心功能：积分获取、积分消费、积分过期...
> 预期收益：DAU提升15%...

**主Agent**：（记录完成）
> 需求陈述完毕，开始分项审议。
>
> 👉 当前审议：**积分获取规则**
>
> 请技术负责人发言。

**技公公**：
> 技术可行性OK。但并发场景下的积分计算需要考虑一致性...
> 建议用消息队列异步处理。

**主Agent**：
> 记录技术建议：消息队列异步处理。
>
> 请设计负责人发言。

**设公公**：
> 用户积分页面需要考虑多端适配...
> 建议PC和移动端分别设计。

**主Agent**：
> 该功能项审议完成，暂无阻塞问题。
>
> 👉 下一项：**积分消费规则**
> ...

**... 依次审议 ...**

**主Agent**：
> 所有功能项审议完毕，进入综合评审。
>
> 汇总：
> ✅ 通过项：5个
> ⚠️ 修改项：1个（积分过期提醒机制需细化）
> ❓ 待定项：0个
>
> 启动整体决议投票。

**... 投票进行 ...**

**主Agent**：
> 投票结果：**修改后通过**（3/3同意）
>
> 行动项：
> 1. [ ] 细化积分过期提醒规则 - 负责人：产公公 - 截止：本周五
> 2. [ ] 技术方案详细设计 - 负责人：技公公 - 截止：下周三
>
> 会议结束，纪要已生成。

## 评审检查清单

### 完整性检查
- [ ] 需求背景和目标明确
- [ ] 功能边界清晰
- [ ] 非功能需求（性能、安全）已定义
- [ ] 验收标准明确

### 可行性检查
- [ ] 技术方案可落地
- [ ] 资源需求合理
- [ ] 时间预期可接受
- [ ] 依赖关系已识别

### 价值检查
- [ ] 用户价值明确
- [ ] 业务收益可量化
- [ ] 与产品定位一致

### 风险检查
- [ ] 技术风险已评估
- [ ] 业务风险已识别
- [ ] 有应对预案
